# Load test — TestNet backend baseline (2026-04-18)

**Subject:** `roil-backend.service` on TestNet VPS (Netcup VPS 500 G12,
2 vCPU, 8 GB RAM), v0.3.3 + rate-limiter patch, hardened systemd
(`/opt/roil-backend`, User=roil).

**Tool:** `autocannon@8.0.0` from local workstation (Turkey), against the
public Caddy TLS endpoint `https://api.roil.app/…`.

**Test window:** 2026-04-18 22:21 – 22:36 UTC+2.

**Setup deltas for the run (reverted after):**
- `RATE_LIMIT_MAX=1000000` in `/opt/roil-backend/.env` so the per-IP
  rate limit (100/min default in `middleware/security.ts`) does not cap
  the synthetic load.
- Rate limiter is env-configurable via `RATE_LIMIT_MAX` starting in this
  build (was hardcoded to 100 previously).

Backend values restored to defaults at end of the test.

## Resource use on the VPS

**During Run 1 (sustained 100 rps on /health):**
- Node process: **6.8% CPU, 97 MB RSS**
- Prometheus `circuit_breaker_state{breaker=ledger}` = 0 (closed)
- Prometheus `circuit_breaker_state{breaker=cantex}` = 0 (closed)
- Histogram count cap at 10,000 working as designed (older samples drop)

Plenty of headroom on a single-vCPU envelope — estimate 10x current rps
before CPU saturation, but `/health` is a cheap endpoint. Real ledger-bound
paths (`/api/portfolio/:party/rebalance`, `/api/dca/*/execute`) would
dominate tail latency through the Canton participant, not Node.

---

## Baseline numbers to watch in Prometheus

| Metric | p50 | p95 | p99 | Note |
|---|---|---|---|---|
| `http_request_duration_ms` (`/health`) | 34 | 66 | 70 | Caddy + Node passthrough |
| `http_request_duration_ms` (`/metrics`) | 34 | 66 | 70 | Prometheus text serialize |

If any future /health p95 exceeds **200 ms** under baseline load, something
has regressed (JSON serialisation changed, participant added to health probe,
network degraded). Alert rule `RoilBackendRequestLatencyP95High` already
fires on p95 > 2000 ms for 10 min — that's an outage-grade threshold;
consider adding a `p95 > 500ms` warning rule after MainNet baseline exists.
